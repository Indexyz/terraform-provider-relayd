# API

`relayd` exposes an authenticated HTTP control-plane API under `/v1`.

## Quick start

```bash
export BASE_URL=http://127.0.0.1:8080
export AUTH_TOKEN=mytoken

curl -sS \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  "${BASE_URL}/v1/ports"
```

## Authentication

All endpoints require a bearer token:

```http
Authorization: Bearer <AUTH_TOKEN>
```

If the header is missing, malformed, or the token does not match, the server returns:

- `401 Unauthorized`
- body: `unauthorized`

## Conventions

- Base path: `/v1`
- Successful `POST`/`GET` responses use `Content-Type: application/json`
- `DELETE /v1/ports/{id}` returns `204 No Content` with an empty body
- Error responses are plain text, not JSON
- Allocation IDs are UUIDv7 strings generated by the server
- `host` accepts IP literals only, for example `127.0.0.1` or `::1`
- `GET /v1/ports` returns allocations ordered by `protocol`, then `port`

## Allocation object

Successful create/update/list responses use this shape:

```json
{
  "id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "protocol": "tcp",
  "port": 10000,
  "target_port": 8080,
  "host": "127.0.0.1",
  "effective_target_port": 8080,
  "effective_host": "127.0.0.1",
  "host_configured": true,
  "runtime_status": "active",
  "error_kind": null,
  "last_error": null,
  "created_at_ms": 1712822400000,
  "updated_at_ms": 1712822405000
}
```

### Field meanings

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | string | Server-generated allocation ID (UUIDv7). |
| `protocol` | `tcp` \| `udp` | Forwarding protocol. |
| `port` | integer | Relay listen port allocated from `PORT_RANGE`. |
| `target_port` | integer | Desired upstream port. |
| `host` | string \| null | Desired upstream IP literal. `null` means not configured yet. |
| `effective_target_port` | integer \| null | Target port currently applied in runtime state. |
| `effective_host` | string \| null | Target host currently applied in runtime state. |
| `host_configured` | boolean | Whether a non-empty desired host is configured. |
| `runtime_status` | string | Runtime state of the listener. |
| `error_kind` | string \| null | Runtime error class when degraded. |
| `last_error` | string \| null | Last runtime error message, if any. |
| `created_at_ms` | integer | Unix timestamp in milliseconds. |
| `updated_at_ms` | integer | Unix timestamp in milliseconds. |

### `runtime_status` values

- `active`
- `rejecting_no_host`
- `degraded_bind_failed`
- `degraded_apply_failed`
- `degraded_close_failed`

### `error_kind` values

- `bind_failed`
- `apply_failed`
- `close_failed`
- `restore_failed`
- `invalid_input`

## Endpoints

### 1. Create allocation

`POST /v1/ports`

Create a relay port reservation for `tcp` or `udp`.

#### Request body

```json
{
  "protocol": "tcp",
  "target_port": 8080
}
```

#### Success

- `201 Created`
- body: one allocation object

#### Notes

- `protocol` must be `tcp` or `udp`
- A new allocation is created without a target host, so the initial `runtime_status` is typically `rejecting_no_host`
- The relay port is picked automatically from `PORT_RANGE`

#### Example

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  -H "Content-Type: application/json" \
  "${BASE_URL}/v1/ports" \
  -d '{"protocol":"tcp","target_port":8080}'
```

### 2. Set target host

`POST /v1/ports/target`

Set or replace the desired upstream host for an existing allocation.

#### Request body

```json
{
  "id": "018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e",
  "host": "127.0.0.1"
}
```

#### Success

- `200 OK`
- body: one allocation object

#### Example

```bash
curl -sS \
  -X POST \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  -H "Content-Type: application/json" \
  "${BASE_URL}/v1/ports/target" \
  -d '{"id":"018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e","host":"127.0.0.1"}'
```

### 3. Update allocation

`POST /v1/ports/{id}`

Patch the desired target port and/or host for an existing allocation.

#### Request body

Any non-empty combination of:

```json
{
  "target_port": 9090,
  "host": "127.0.0.1"
}
```

#### Success

- `200 OK`
- body: one allocation object

#### Validation

- At least one of `target_port` or `host` must be present
- Empty body `{}` returns `400 Bad Request` with body `empty update`

### 4. Delete allocation

`DELETE /v1/ports/{id}`

Remove the allocation and stop its runtime listener.

#### Success

- `204 No Content`
- empty body

#### Example

```bash
curl -sS \
  -X DELETE \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  "${BASE_URL}/v1/ports/018f0d5d-86d6-7e57-8bb5-c0b7c38e2f0e"
```

### 5. List allocations

`GET /v1/ports`

Return all allocations.

#### Success

- `200 OK`
- body: array of allocation objects

#### Example

```bash
curl -sS \
  -H "Authorization: Bearer ${AUTH_TOKEN}" \
  "${BASE_URL}/v1/ports"
```

## Error mapping

These are the current HTTP-level behaviors implemented by the server.

| Condition | Status | Body |
| --- | --- | --- |
| Missing/bad bearer token | `401` | `unauthorized` |
| Unknown route | `404` | `not found` |
| Unknown allocation ID | `404` | `NotFound` |
| Invalid `protocol` | `400` | `invalid protocol` |
| Invalid `host` IP literal | `400` | `InvalidHost` |
| Empty update body (`{}`) | `400` | `empty update` |
| No free port left in `PORT_RANGE` | `409` | `NoAvailablePort` |
| Runtime apply/delete timeout or runtime update failure | `503` | `Timeout`, `RuntimeUpdateFailed`, or `RuntimeDeleteFailed` |
| Other unhandled server errors | `500` | plain-text error or `internal server error` |

## Operational notes

- Creating an allocation does not make it immediately usable unless a target host is configured.
- Successful create/update responses return the latest allocation view, including desired fields (`host`, `target_port`) and runtime-applied fields (`effective_host`, `effective_target_port`).
- This API currently uses `POST` for updates instead of `PATCH`.
- Malformed JSON is not normalized into a structured `400` response today; depending on where parsing fails it may surface as `500 internal server error`.
